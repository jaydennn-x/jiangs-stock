# JiangsStock MVP PRD

## 핵심 정보

**목적**: 스톡 이미지를 크기별·라이선스별로 판매하고, 구매자에게 사이트 내 보안 다운로드를 통해 디지털 파일을 안전하게 전달하는 1인 개발 이커머스 플랫폼 구축

**사용자**: 블로그·웹사이트·마케팅 자료용 고품질 이미지를 구매하는 개인/기업, 그리고 이미지 업로드·가격 설정·주문 관리를 수행하는 1인 운영자(관리자)

전체적으로 기술 스택 선정이 합리적이며 1인 개발 MVP로서 적절한 범위를 유지하고 있습니다.

- **기술적 신뢰도**: 8/10

### Critical Issues (반영 완료)

**C1. 보안 취약점 - Next.js 미들웨어 우회 (CVE-2025-29927)**

- [FACT] Next.js 15.2.3 미만 버전에서 x-middleware-subrequest 헤더를 통한 미들웨어 인증 우회 가능 (CVSS 9.1)
- 현재 프로젝트는 Next.js 15.5.3을 사용하므로 이미 패치된 버전이나, 미들웨어 기반 인증에만 의존하지 않도록 Server Action/API Route 레벨에서도 인증 검증 필수
- **조치**: 보안 요구사항 섹션에 다중 레이어 인증 검증 명시

**C2. NextAuth.js v5 베타 상태**

- [FACT] Auth.js v5는 2023년 10월 beta.0 출시 이후 지속적으로 베타 상태이며, 정식 GA 일정이 발표되지 않음
- [INFERENCE] 그러나 프로덕션에서 광범위하게 사용되고 있으며, Next.js 15/16과의 호환성이 확인됨
- **조치**: 기술 스택에 베타 상태를 명시하고, @auth/prisma-adapter 호환성 확인 필요 표기

**C3. Zod 버전 불일치**

- [FACT] package.json에는 Zod 4.x (^4.1.11)가 설치되어 있으나, PRD에는 "Zod 3.x"로 기재됨
- Zod 4.x는 API 변경이 있으므로 정확한 버전 기재 필요
- **조치**: 기술 스택의 Zod 버전을 4.x로 수정

**C4. Sharp 버전 업데이트**

- [FACT] Sharp 최신 버전은 0.34.x이며, PRD에는 0.33.x로 기재됨
- **조치**: 기술 스택의 Sharp 버전을 0.34.x로 수정

### Major Issues (반영 완료)

**M1. 데이터 모델 누락 - 결제 검증 및 멱등성**

- Order 모델에 결제 검증을 위한 idempotencyKey(멱등성 키) 필드 누락
- PG사 웹훅 수신 시 중복 처리 방지를 위해 필수
- **조치**: Order 모델에 idempotencyKey 필드 추가

**M2. 데이터 모델 누락 - 이미지 파일 크기 정보**

- Image 모델에 각 크기별 파일 크기(bytes) 정보 필드 누락
- F002(이미지 상세 보기)에서 "파일 크기" 메타데이터 표시가 요구되나, 이를 저장할 필드가 없음
- **조치**: Image 모델에 fileSizesJson 필드 추가

**M3. 데이터 모델 누락 - 해상도 정보**

- Image 모델에 원본 해상도(width, height) 필드 누락
- F003에서 각 크기별 해상도 표시가 요구되며, 크기별 비율 계산의 기준값 필요
- **조치**: Image 모델에 width, height 필드 추가

**M4. 장바구니 세션 전략 미정의**

- F004에서 "세션 기반 비로그인 장바구니"를 명시하나, 구체적 저장 전략이 없음
- [INFERENCE] Zustand + localStorage persist가 가장 실용적인 선택
- 로그인 시 로컬 장바구니를 서버와 병합하는 전략도 필요
- **조치**: 장바구니 저장 전략 명시

**M5. 검색 기능 구현 전략 미정의**

- F001에서 키워드 검색을 명시하나, 검색 엔진 또는 DB 검색 전략이 없음
- [FACT] PostgreSQL 16은 tsvector/tsquery 기반 전문 검색을 네이티브 지원
- **조치**: PostgreSQL Full-Text Search + trigram 인덱스 전략 명시

**M6. 결제 웹훅 처리 누락**

- F005에서 PG사 콜백 리디렉션만 언급하나, 서버 간 웹훅 수신 처리가 누락됨
- [FACT] 포트원 v2는 결제 완료 후 웹훅 알림을 전송하며, 이를 수신하여 주문 상태를 확정하는 것이 필수
- **조치**: 웹훅 수신 API 엔드포인트 및 처리 흐름 명시

**M7. 기능 ID 번호 불연속**

- F008 다음이 F010으로 건너뛰어 F009가 빠져 있음
- **조치**: 가독성을 위해 현행 유지하되, 향후 확장을 위한 예약 번호로 표기

**M8. 다운로드 전략 일관성**

- F006에서 "S3 Presigned URL"을 명시하고, F017에서 "CloudFront Signed URL"을 명시
- [FACT] 두 가지를 병행 사용할 수 있으나, 역할 분리가 필요: 프리뷰/썸네일은 CloudFront, 다운로드는 S3 Presigned URL
- **조치**: 로컬 파일 서빙으로 단일화되어 해소 — API Route 스트리밍(다운로드) + 단순 API Route(프리뷰) + API Route 서빙(썸네일)로 역할 구분

**M9. 이미지 처리 비동기화 미정의**

- F014에서 Sharp를 이용한 이미지 리사이징을 업로드 시 동기적으로 처리하는 것으로 기재됨
- [INFERENCE] 고해상도 이미지(원본 30MB 이상) 다량 업로드 시 Sharp 처리가 메인 스레드를 블로킹하여 CPU 과부하 및 서버 응답 불가 상태 발생 위험
- [FACT] Node.js는 단일 스레드 기반이므로 CPU 집약적 작업(리사이징, 워터마크 합성)을 메인 이벤트 루프에서 처리하면 전체 서비스가 영향을 받음
- **조치**: BullMQ + Redis 기반 비동기 대기열 도입. 업로드는 즉시 완료 처리, 이미지 리사이징/워터마크 합성은 백그라운드 워커에서 비동기 처리

**M10. 다운로드 링크 남용 방어**

- UUID 토큰 기반 보안 링크는 3회·7일 만료로 충분한 억제력 확보
- IP/UA 로깅은 TransactionLog에 기록, 실제 남용 패턴 관측 후 차단 로직 도입
- **조치**: 다운로드 횟수 제한(3회·7일)만 적용. IP/UA는 TransactionLog에 기록만 하고 적극 차단 로직은 MVP 이후 도입

### Minor Suggestions

**S1. 이메일 인증 미포함**: MVP에서 이메일 인증(가입 확인 메일) 미포함. SES가 이미 스택에 있으므로 낮은 추가 비용으로 구현 가능. MVP 이후 고려 권장.

**S2. 비밀번호 재설정 미포함**: F010(인증)에 비밀번호 찾기/재설정 기능이 없음. 필수 기능은 아니나 MVP에서도 포함하는 것이 일반적. MVP 이후 고려 권장.

**S3. Rate Limiting 미정의**: API 엔드포인트에 대한 요청 제한 정책이 없음. 로그인 시도 제한, 다운로드 요청 제한 등 기본적인 Rate Limiting 필요.

**S4. 이미지 업로드 크기 제한 미정의**: F013/F014에서 원본 이미지 업로드의 최대 파일 크기가 미정의. Next.js의 기본 body 크기 제한(4MB)을 고려하여 Next.js body 크기 제한 조정(50MB) + Nginx client_max_body_size 설정 필요.

**S5. 매출 추이 차트 라이브러리 미정의**: F016에서 차트를 요구하나 차트 라이브러리가 기술 스택에 없음. Recharts 또는 Chart.js 추가 권장.

## 보안 요구사항

### 인증 보안

- **다중 레이어 인증 검증**: 미들웨어 인증 외에, Server Action 및 API Route 핸들러 내부에서도 세션/권한을 반드시 재검증
- **관리자 경로 보호**: /admin 경로는 미들웨어 + Server Action 양쪽에서 ADMIN 역할 검증
- **비밀번호 정책**: 최소 8자, 영문+숫자+특수문자 조합, bcryptjs saltRounds 12 이상
- **세션 관리**: JWT 기반 세션 (httpOnly + secure + sameSite=strict 쿠키). **강제 로그아웃 지원**: Redis에 세션 블록리스트를 관리하여 관리자 계정 탈취 시 토큰 만료 전 즉시 무효화 가능
- **관리자 IP 화이트리스트**: 관리자 로그인 시 `ADMIN_ALLOWED_IPS` 환경변수에 등록된 IP 대역에서만 허용. 미등록 IP 접근 시 즉시 거부 (401 응답). CIDR 표기 지원 (예: `1.2.3.4/32,5.6.7.8/32`)
- **관리자 로그인 알림 (MVP)**: 관리자 계정 로그인 성공 시 AWS SES로 운영자 이메일에 즉시 알림 발송 (IP, User-Agent, 시각 포함). 침입 탐지의 최소 안전망으로 MVP에 포함.
- **관리자 2FA (MVP 이후)**: 관리자 계정 탈취 시 모든 원본 자산이 유출될 수 있는 단일 실패 지점 위험. MVP 이후 TOTP 기반 2FA 또는 OTP 인증 도입 예정.

### 인프라 보안 (AWS 보안 그룹)

- **EC2 인바운드**: 80/443 포트는 ALB 보안 그룹에서만 허용 (외부 직접 접근 차단). SSH(22)는 특정 관리자 IP만 허용
- **RDS 인바운드**: 5432 포트는 EC2 보안 그룹에서만 허용
- **ALB 인바운드**: 80/443은 0.0.0.0/0 허용 (인터넷 공개)
- **ElastiCache 인바운드**: 6379 포트는 EC2 보안 그룹에서만 허용

### API 보안

- **HTTPS/TLS**: ALB + ACM 인증서로 HTTPS 적용 필수. 결제 처리 포함 서비스에서 미적용 불가
- **CORS 정책**: 결제 콜백·웹훅 수신 엔드포인트는 PG사 도메인/IP만 허용. 일반 API는 서비스 도메인만 허용
- **CSRF 보호**: NextAuth.js v5 내장 CSRF 보호 활용
- **Rate Limiting**: 로그인 시도 (5회/분), 결제 요청 (3회/분), 다운로드 요청 (10회/분)
- **입력 검증**: 모든 API 엔드포인트에서 Zod 스키마로 입력값 검증
- **SQL Injection 방지**: Prisma ORM의 파라미터 바인딩 활용
- **헬스체크 엔드포인트**: `/api/health` — DB 연결, Redis 연결, 디스크 공간 체크. CloudWatch 모니터링 연동용

### 파일 보안

- **썸네일**: API Route 서빙 (`/api/images/thumbnail/{imageId}`), 인증 불필요, 저해상도 공개 — `public/` 복사 방식 사용 금지 (배포 시 데이터 유실 위험). `Cache-Control: public, max-age=86400, immutable` 헤더로 브라우저/CDN 캐싱 활성화
- **워터마크 프리뷰**: 단순 API Route (`/api/images/preview/{imageId}`), 인증 불필요. 워터마크 자체가 원본 보호 수단이며 추가 서명 불필요
- **다운로드 파일 (On-Demand 생성)**: 서버에 고품질 파일 상시 미보관. 다운로드 요청 시 Worker가 로컬 하드 원본에서 리사이즈 생성 → AES-256-GCM 암호화 임시 저장 → Next.js가 복호화 스트림 전송 → 즉시 삭제. **프로세스 권한 분리**: 웹서버(Next.js, webuser 계정)는 원본 이미지 디렉토리 접근 불가, Worker(worker 계정)만 읽기 전용 접근. UUID 토큰 검증 + 6중 보안 검증 (토큰 유효성, 만료, 횟수, 소유자, Rate Limit, 주문 상태). 응답에 `Referrer-Policy: no-referrer`, `Cache-Control: no-store` 헤더 적용. 다운로드 IP·UA는 TransactionLog에 기록. 상세: `docs/guides/secure-download-architecture.md` 참조
- **원본 파일**: OS 레벨 권한(chmod 700, worker 계정 전용) + 관리자 API Route 인증 (`/api/admin/images/{imageId}/original`, 세션 + ADMIN 역할 검증). 웹서버 프로세스에서 직접 접근 불가
- **Path Traversal 방어**: imageId는 API 진입 시 UUID v4 정규식(`/^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i`)으로 검증. 파일 읽기 전 `path.resolve(STORAGE_ROOT, resolvedPath)`가 반드시 `STORAGE_ROOT` 하위에 있는지 확인 후 거절.
- **다운로드 원자적 업데이트**: 다운로드 횟수 증가 시 Prisma `data: { downloadCount: { increment: 1 } }` 원자적 업데이트 사용. DB 레벨에서 순차 처리하여 동시 요청에 의한 횟수 우회(Race Condition) 방지.
- **우클릭/드래그 방지**: 클라이언트 측 보호 (완벽한 보호는 아니나 일반 사용자 수준 방지)

### 결제 보안

- **서버 사이드 금액 검증**: 클라이언트에서 전달된 금액을 서버에서 재계산하여 일치 여부 확인 후 결제 승인. SystemConfig의 크기별 비율 및 라이선스 배율 기준으로 서버에서 독립 계산
- **웹훅 서명 검증 + IP 화이트리스트**: PG사 웹훅 수신 시 서명(signature) 검증 + PG사 공개 IP 대역 화이트리스트 적용. 서명만으로는 리플레이 공격 방어 불완전
- **웹훅 멱등성 가드**: 웹훅 수신 시 주문 상태가 이미 COMPLETED이면 즉시 200 응답 후 처리 스킵. 동일 웹훅 중복 발송에 의한 이중 처리 방지
- **멱등성 보장**: idempotencyKey를 활용한 중복 결제 방지

## 캐싱 전략

Redis가 이미 스택에 있으므로 애플리케이션 레벨 캐싱 적극 활용:

| 캐시 대상              | TTL   | 이유                            |
| ---------------------- | ----- | ------------------------------- |
| 카테고리 목록          | 1시간 | 거의 변경 없음                  |
| 랜딩 베스트셀러 TOP 12 | 10분  | 판매 건수 변동 주기             |
| 검색 결과 (동일 쿼리)  | 5분   | DB 부하 감소                    |
| SystemConfig 전체      | 5분   | 설정 변경 즉시 반영 필요성 낮음 |

**HTTP 응답 캐시 헤더 정책**:

- 썸네일: `Cache-Control: public, max-age=86400, immutable`
- 워터마크 프리뷰: `Cache-Control: public, max-age=3600` (인증 불필요한 공개 리소스, 워터마크가 원본 보호 수단이므로 캐싱 허용. `private`는 공개 리소스에 의미상 부적절하며 Node.js 매번 파일 읽기로 성능 불리)
- 다운로드: `Cache-Control: private, no-store`
- 정적 자산(JS/CSS): `Cache-Control: public, max-age=31536000, immutable`

## 검색 전략

### PostgreSQL Full-Text Search

- **검색 대상**: Image.name, Image.description, Image.tags
- **구현 방식**: PostgreSQL tsvector/tsquery + GIN 인덱스
- **한국어 지원**: pg_bigm 또는 trigram (pg_trgm) 확장으로 한국어 부분 일치 검색 지원
- **인덱스 전략**:
  - tags 배열: GIN 인덱스
  - colorTags 배열: GIN 인덱스
  - name + description: tsvector GIN 인덱스
  - categoryId + isActive: B-tree 복합 인덱스
- **정렬**: createdAt(최신순), salesCount(인기순), basePrice(가격순)

## 장바구니 전략

- **비로그인 상태**: Zustand + localStorage persist로 클라이언트 측 저장
- **로그인 상태**: 서버 DB의 Cart/CartItem 모델 사용 (Server Action으로 동기화)
- **로그인 전환 시 병합 처리**:
  1. 로그인 완료 후 로컬 장바구니 데이터를 Server Action으로 전송
  2. 서버의 기존 Cart와 병합: 동일한 imageId + size + licenseType 조합은 로컬 항목 우선 적용, 그 외 항목은 합산
  3. 병합 완료 후 서버에서 현재 가격으로 재계산. 가격이 변경된 아이템은 사용자에게 알림 표시 ("일부 상품의 가격이 변경되었습니다")
  4. 병합 완료 후 로컬 localStorage 장바구니 초기화
- **병합 규칙**: 충돌 시 로컬 데이터 우선 (사용자가 방금 선택한 옵션 존중)
- **장바구니 아이템 구조**: imageId, size, licenseType, price (가격은 결제 시 서버에서 재검증)

## 결제 처리 흐름

```
1. [클라이언트] 결제 요청 --> [서버] 주문 생성 (PENDING) + 금액 서버 계산 (SystemConfig 기준)
2. [서버] PG사 결제 요청 (idempotencyKey 포함) --> [PG사] 결제 처리
3. [PG사] 결제 결과 리디렉션 --> [클라이언트] 결제 완료 페이지
4. [PG사] 웹훅 --> [서버] /api/webhooks/payment (IP 화이트리스트 + 서명 검증)
   |
   v
5. [서버] 멱등성 가드: 주문 상태 COMPLETED면 즉시 200 응답 후 스킵
   --> 주문 상태 PENDING이면: 결제 금액 일치 확인 --> 주문 상태 COMPLETED 변경
   |
   v
6. [서버] 다운로드 토큰 생성 (OrderItem별 UUID, 3회 한도·7일 만료)
   + BullMQ에 구매 확인 알림 이메일 발송 작업 등록 (비동기, 다운로드 링크 미포함)
   --> 이메일은 구매 확인 알림 목적만 (다운로드는 사이트 내 마이페이지에서만 가능)
   --> 이메일 발송 실패 시 자동 재시도(최대 3회, 지수 백오프) + TransactionLog FAILURE 기록
   --> 이메일 실패가 주문 상태에 영향을 주지 않음
   |
   v
7. [구매자] 결제 완료 페이지 또는 마이페이지 → 다운로드 버튼 클릭
   --> [서버] 6중 보안 검증 (토큰·만료·횟수·소유자·Rate Limit·주문 상태)
   --> [Worker] 로컬 하드 원본에서 On-Demand 리사이즈 생성 → 암호화 임시 저장
   --> [서버] 복호화 → 스트림 전송 → 임시 파일 즉시 삭제
   --> 서버에 고품질 파일 상시 미보관 (상세: docs/guides/secure-download-architecture.md)
```

**핵심 원칙**:
- 결제 완료 판정은 클라이언트 리디렉션이 아닌, 서버 웹훅 수신을 기준으로 확정
- 다운로드는 사이트 내 전용 (이메일 첨부 발송 없음), 서버에 고품질 파일 상시 미보관

### 저장 구조 (3-Tier 보안 아키텍처)

> 상세 설계: `docs/guides/secure-download-architecture.md` 참조

```
[Tier 1: 서버 상시 보관 - browsing용, 해킹당해도 상업 가치 없음]
/data/jiangs-storage/           # EC2 EBS 마운트 (프로젝트 외부)
├── watermarks/{imageId}/       # 워터마크 프리뷰 (API Route 서빙)
├── thumbnails/{imageId}/       # 썸네일 (API Route 서빙, public/ 복사 불필요)
└── pending-downloads/          # 임시 암호화 파일 (다운로드 요청 시 생성, 전송 후 삭제)

[Tier 2: 로컬 하드디스크 - 원본, 서버 프로세스 접근 불가]
/mnt/originals/                 # 프로덕션 (chmod 700, worker 계정 전용)
D:\jiangsstock\                 # 개발 환경
├── IMG_0001.jpg
└── ...

[Tier 3: 임시 다운로드 파일 - 구매 시 On-Demand 생성, 전송 후 즉시 삭제]
pending-downloads/{hash}/       # AES-256-GCM 암호화, 키는 Redis(TTL 5분)
```

> **핵심 원칙**: 서버에 상업적 가치가 있는 고품질 이미지를 상시 보관하지 않습니다. 다운로드 요청 시에만 Worker가 로컬 하드 원본에서 On-Demand 리사이즈 생성 → 암호화 임시 저장 → 전송 후 즉시 삭제합니다.
> **주의**: 썸네일을 `public/` 폴더로 복사하지 않습니다. 배포(빌드) 시 `public/`이 초기화되어 파일이 유실될 수 있으며, 코드(배포 아티팩트)와 데이터(이미지 파일)를 물리적으로 분리해야 합니다.

| -------------------- | ------------------ | --------------------------------------------------------- | --------------------------------------------- |

### 환경변수

| 변수                | 설명                                                                                                           |
| ------------------- | -------------------------------------------------------------------------------------------------------------- |
| `STORAGE_ROOT`      | 이미지 저장 루트 경로 (예: `/data/jiangs-storage`)                                                             |
| `ORIGINALS_ROOT`    | 원본 이미지 경로 (예: `/mnt/originals` 또는 `D:\jiangsstock`). Worker 프로세스만 접근 가능                     |
| `SHARP_CONCURRENCY` | Sharp 동시 처리 수 (기본 2, 인스턴스 코어 수에 맞게 조정)                                                      |
| `ADMIN_ALLOWED_IPS` | 관리자 로그인 허용 IP 대역 (CIDR 쉼표 구분, 예: `1.2.3.4/32,5.6.7.8/32`). 미설정 시 모든 IP 허용 (개발 환경용) |
